# API внешнего источника данных для Hockey Scoreboard Host

**Оглавление документации:** [docs/README.md](README.md)

Этот документ описывает контракт HTTP API, который должен реализовать **другой сервис** (vmix-proxy, бэкенд арены, самописный Node/Python сервер и т.д.), чтобы **приложение Hockey Desktop Host** (Tauri + Rust gateway) могло опрашивать его и обновлять табло в OBS.

Локальный gateway приложения **не является** этим API: он только проксирует опрос вашего URL и раздаёт оверлей по `http://127.0.0.1:<порт>/`.

---

## 1. Как клиент обращается к вашему серверу

| Параметр | Значение |
|----------|----------|
| **HTTP-метод** | `GET` |
| **URL** | Любой полный URL, который пользователь вводит в поле «URL внешнего API» (например `https://stats.example.com/api/vmix` или `http://192.168.1.10:8080/api/vmix`). |
| **Тело запроса** | Нет (GET без body). |
| **Заголовки со стороны клиента** | Стандартные для `reqwest`: клиент не добавляет специальных заголовков авторизации из UI. При необходимости (Bearer, API-key) их нужно либо вшить в URL (редко), либо доработать клиент в коде — из коробки только «голый» GET. |
| **Интервал опроса** | **~800 мс** между успешными попытками цикла (фиксированный `interval` в Rust). Первый тик после старта тоже с задержкой в рамках этого интервала. |
| **Таймаут одного запроса** | **5 секунд**. При таймауте или сетевой ошибке текущее состояние табло **не обновляется** (остаётся последнее успешное или дефолтное). |
| **Успешный ответ** | HTTP статус **2xx** (`is_success()` в `reqwest`). Иначе тело **не парсится**, состояние не меняется. |
| **Формат тела ответа** | JSON: объект **или** массив JSON (см. ниже). Тип `Content-Type` должен позволить клиенту распарсить JSON; обычно достаточно `application/json`. |

**CORS** для этого запроса не важен: запрос выполняется из **нативного** приложения (Rust), а не из браузера.

## 2. Формат JSON-ответа

### 2.1. Объект (рекомендуется)

Корень ответа — **JSON-объект**. Он трактуется как **частичное обновление** поверх встроенного состояния по умолчанию (см. §3).

```json
{
  "ScoreA": 3,
  "ScoreB": 2,
  "Timer": "15:42"
}
```

Все **отсутствующие** поля заполняются **дефолтами** (как в таблице §3).

### 2.2. Массив (совместимость, например как у некоторых vmix-подобных ответов)

Если корень — **массив**, клиент берёт **только первый элемент** (`raw[0]`). Если это объект — с ним делается такое же **поверхностное (shallow) слияние** с дефолтами (удобно для форматов вроде одиночного элемента в массиве).

```json
[
  {
    "TeamA": "ЦСКА",
    "TeamB": "СКА",
    "ScoreA": 1,
    "ScoreB": 0
  }
]
```

- Если массив **пустой** — в слияние идёт `null` → фактически остаются одни дефолты.
- Если первый элемент **не объект** — поведение неоднозначно; передавайте объект.

### 2.3. Поверхностное слияние (shallow merge)

- Берётся объект **дефолтов** (полный набор ключей).
- Каждый **ключ из вашего объекта** **перезаписывает** соответствующий ключ дефолта **одним уровнем**.
- **Вложенные** структуры из вашего API в состоянии табло **не используются**: ожидается **плоский** объект полей `GameState`. Если вы передадите вложенный JSON одним значением ключа, он уйдёт в `serde_json::Value` как есть — фронт оверлея ожидает примитивы/строки по контракту ниже.

**Дополнительные ключи** (не из таблицы) тоже попадут в общий объект состояния после merge; компоненты табло могут их игнорировать. Лучше не слать лишнее, чтобы не путать отладку.

## 3. Поля состояния (`GameState`)

Имена полей **регистрозависимы** (как в TypeScript-интерфейсе): например `TeamA`, а не `teama`.

| Поле | Тип | Значение по умолчанию | Описание |
|------|-----|------------------------|----------|
| `TournamentTitle` | `string` | `"Регулярный турнир по хоккею с шайбой"` | Левая строка синей верхней панели (название турнира / лиги). |
| `SeriesInfo` | `string` | `""` | Правая строка верхней панели (серия, счёт серии и т.п.). |
| `BrandingImage` | `string` | `""` | Имя файла картинки брендинга: ожидается использование с базой логотипов в оверлее (`logos/…`). Пустая строка — показывается текст по умолчанию оверлея. |
| `TeamA` | `string` | `"A"` | Короткое имя команды A. |
| `TeamAFull` | `string` | `"Team A"` | Полное имя команды A. |
| `TeamB` | `string` | `"B"` | Короткое имя команды B. |
| `TeamBFull` | `string` | `"Team B"` | Полное имя команды B. |
| `penalty_a` | `string` | `"None"` | Отображение штрафа/большинства для команды A (строка, семантика зависит от оверлея). |
| `penalty_b` | `string` | `"None"` | То же для команды B. |
| `ScoreA` | `number` | `0` | Голы команды A. |
| `ScoreB` | `number` | `0` | Голы команды B. |
| `ShotsA` | `number` | `0` | Броски команды A. |
| `ShotsB` | `number` | `0` | Броски команды B. |
| `logo_a` | `string` | `"team-a.png"` | Имя файла логотипа A в каталоге логотипов оверлея. |
| `logo_b` | `string` | `"team-b.png"` | Имя файла логотипа B. |
| `Timer` | `string` | `"20:00"` | Основной таймер, формат **`MM:SS`** (как строка). |
| `PowerPlayTimer` | `string` | `"02:00"` | Таймер большинства (**`MM:SS`**). |
| `PowerPlayActive` | `boolean` | `false` | Активно ли большинство (влияет на отображение/логику таймеров в оверлее). |
| `Period` | `number` | `1` | Номер периода (целое). |
| `Running` | `boolean` | `false` | Идёт ли игра с точки зрения таймера. |
| `Visible` | `boolean` | `true` | Видимость табло. |

Числа и булевы в JSON должны быть **настоящими** JSON number/boolean, не строками `"true"` / `"1"`, если вы хотите корректную семантику во фронте (иначе тип может не совпасть с ожиданиями React).

## 4. Поведение при ошибках и краевые случаи

| Ситуация | Результат |
|----------|-----------|
| Не 2xx | Тело не разбирается; состояние **не обновляется**. |
| Таймаут 5 с или сетевая ошибка | Как выше. |
| 200, но **не JSON** | Парсинг падает; обновления нет. |
| 200, пустое тело | Обычно ошибка парсинга; обновления нет. |
| Первый успешный ответ после старта | Состояние заменяется на `merge(дефолт, ваш_объект)`. |
| Режим «Тест» в приложении | Внешний API **не опрашивается**; этот контракт не используется. |

## 6. Проверка из командной строки

Замените URL на ваш реальный эндпоинт:

```bash
curl -sS -D - "http://127.0.0.1:8080/api/vmix" -o /tmp/state.json
jq . /tmp/state.json
```

Убедитесь, что статус **2xx** и тело — валидный JSON объект или массив, как описано выше.

## 7. Связь с локальным gateway (что видит OBS)

После запуска сервера в приложении пользователь получает локальный URL, например:

- Страница оверлея: `http://127.0.0.1:8787/` (или другой выбранный порт).
- Текущее **уже смёрженное** состояние: `GET http://127.0.0.1:8787/api/state` — тот же JSON, что ушёл в оверлей после последнего успешного опроса **вашего** API.
- WebSocket: `ws://127.0.0.1:8787/ws`. Сообщения: текст JSON вида  
  `{"type":"state","payload":{ ... GameState ... }}`  
  при каждом успешном обновлении с внешнего API.

Внешнему API **не нужно** знать про эти пути — только ваш статический JSON по вашему URL.

## 8. Рекомендации по проектированию сервера

1. **Идемпотентность GET**: клиент дергает URL очень часто (~1.25 Гц). Ответ должен быть быстрым (< 5 с) и не менять серверное состояние там, где это нежелательно.
2. **Кэширование**: если тяжёлая логика — кэшируйте снимок табло в памяти и отдавайте GET из кэша.
3. **Авторизация**: если нужна, сейчас штатный UI её не задаёт; варианты — секрет в query string, IP-фильтрация на вашей стороне, или доработка Rust-клиента (заголовки).
4. **HTTPS**: поддерживается, если система доверяет сертификату (`rustls`).
5. **Согласованность с превью**: поле «Проверить данные» в приложении делает **другой** запрос — обычный `fetch` из WebView к тому же URL; для кросс-доменного URL возможны **CORS-ошибки только в превью**, тогда как **опрос из Rust** продолжит работать. Имейте в виду при отладке.

## 9. Соответствие коду репозитория

- Дефолты и merge: `apps/desktop-host/src-tauri/src/gateway.rs` (`default_state_value`, `merge_external_payload`).
- Цикл опроса: `poll_loop` (интервал **800 ms**, таймаут **5 s**).
- Типы для фронта: `packages/shared/types/gameState.ts`.
- Превью в UI: `apps/desktop-host/src/shared/parseExternalState.ts` (логика «массив или объект» + spread с `defaultGameState` должна совпадать по смыслу с Rust).

При изменении набора полей обновляйте **и** Rust, и `gameState.ts`, и этот документ.
